# Personalized Event Tracker — Product Requirements Document

**Status:** Draft v2 · **Date:** 2026-06-08 · **Author:** Alexander Foltas
**Context:** Sprint 3 portfolio project for an AI Agents course. Local-only MVP; designed for extension.
**Companion document:** `docs/specs/2026-06-08-event-tracker-tech-design.md` covers stack, repo layout, data flow, and other implementation detail.

---

## 1. Overview

Event Tracker is a personal AI agent that learns a user's taste in events through lightweight feedback and surfaces a small, justified daily pick of things worth their time in Hamburg, backed by a chat assistant that can refine, search, and plan.

The MVP runs locally and is built for a single user (the author), with every architectural decision keeping a multi-user Phase 2 mechanical to add.

---

## 2. Problem statement

People in cities miss out on events they would love because event discovery is:

- **Fragmented.** Information lives across municipal sites, ticket platforms, social feeds, and word of mouth.
- **Impersonal.** Generic event listings ignore individual taste — even when filters exist, they do not learn from how you actually react to things.
- **Noisy.** Algorithmic feeds optimise for engagement, not personal fit.
- **Effortful.** Users have to actively go looking, evaluate dozens of options, and remember what they saw.

The result is weekends spent on autopilot while a perfect event happens two streets away.

---

## 3. Target user

**MVP user.** A single user — the author — living in Hamburg with personal interests across music, tech, art, food, and outdoor activities, who wants relevant, curated event suggestions delivered passively and refinable through conversation.

**Phase 2 user.** Any city dweller who wants the same experience for their own city and tastes. The architecture treats `user_id` as a first-class concept from day one so this expansion does not require a rewrite.

---

## 4. Product vision

> *A personal AI agent that learns your taste in events through lightweight feedback and surfaces a small, justified daily pick of things worth your time — with a chat assistant that can refine, search, and plan.*

The agent is the product. The dashboard, calendar, and chat are surfaces through which the agent acts. What distinguishes Event Tracker from a recommender system is that the LLM **reasons** about the user's taste, **justifies** its picks in natural language, **decides** which tools to invoke for a given request, and **remembers** preferences across sessions.

---

## 5. Goals and non-goals

### Goals (MVP)

1. Deliver a daily, personalised, justified digest of 3–5 Hamburg events on app open.
2. Let the user refine recommendations through a chat assistant that has access to events, profile, feedback history, and calendar.
3. Improve recommendations measurably from one day to the next based on thumbs-up / thumbs-down feedback **and optional free-text comments on each event** ("loved the venue, hated the lineup"). The agent uses these comments alongside the binary signal when updating its model of the user.
4. Provide a browsable feed and a calendar view of saved events.
5. Run entirely locally with one command per service.

### Non-goals (MVP)

- Multi-user accounts and authentication.
- Email or push notifications.
- Cities beyond Hamburg.
- Native mobile app.
- Event ticketing or purchase flow (the app links out to the source).
- Cloud deployment.

## 6. MVP feature scope

| # | Feature                                                                                                                                            | Notes                                                                                      |
| - | -------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------ |
| 1 | **Onboarding** — city (Hamburg, fixed), interest tags (multi-select), optional free-text "about me"                                         | Stored in profile; injected into agent prompts.                                            |
| 2 | **Event ingestion** — Eventbrite + Ticketmaster APIs + one Hamburg-specific scraper, normalised to a shared schema                          | Runs once per day.                                                                         |
| 3 | **Daily on-open digest** — 3–5 picks at the top of the dashboard with per-card natural-language justification                              | Generated by the agent in*curation mode*; cached for the day.                            |
| 4 | **Browsable feed** — paginated, filterable list of upcoming events with thumbs-up / thumbs-down and an optional comment field on each card  | Below the digest on the dashboard.                                                         |
| 5 | **Chat assist** — side panel for refinement, search, follow-ups, and actions                                                                | Conversational mode of the same agent; streamed responses.                                 |
| 6 | **Calendar view** — month/week view of events the user has marked interested / saved                                                        |                                                                                            |
| 7 | **Feedback loop** — every thumbs-up/down (with an optional free-text comment) updates the user's taste profile and the recommendation model | Visibly improves the next day's digest. Comments feed the agent's distilled taste summary. |

## 7. Key user flows

1. **First-time setup.** Land on app → onboarding form (city, interests, free-text) → dashboard with first digest already loaded.
2. **Daily use.** Open app → see today's digest with justified picks → react (optionally with a comment) → optionally chat to refine → optionally save events to calendar.
3. **Active discovery.** Scroll browsable feed → filter by date / category → react on cards (with optional comment) → save interesting ones.
4. **Chat-driven planning.** Ask the assistant something concrete ("what's on Friday evening that's cheap and outdoorsy?") → the agent uses tools to find and justify a shortlist → the user saves picks by click or chat command.

---

## 8. Agent behaviour (product view)

What the agent does, from the user's perspective:

- **Two modes.** *Curation mode* runs once per day on first open and produces the digest. *Conversational mode* runs on every chat message.
- **Tool use.** The agent has access to event search, personalised recommendation, feedback recording, calendar read/write, and profile read/write. The settings page exposes per-tool enable/disable toggles so the user can constrain what the agent is allowed to do.
- **Memory.** The agent remembers the conversation within a session and persistent facts across sessions: profile, every reaction (with comments), saved events, and a continuously updated natural-language summary of the user's taste.
- **Justification.** Every recommendation in the digest is accompanied by a one-paragraph explanation grounded in the user's interests, history, and (where relevant) comments.
- **Cold start.** With no feedback yet, recommendations are derived purely from stated interests.

Implementation of these behaviours — tool signatures, memory storage, retrieval logic, model choices — is covered in the technical design document.

## 10. Product risks

| Risk                                                     | Mitigation                                                                                        |
| -------------------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| Sparse coverage of small / indie venues from API sources | Hamburg scraper supplements; the app remains usable on APIs alone.                                |
| Scraper legal or maintenance fragility                   | Pick a source with permissive terms; treat the scraper as best-effort; document limitations.      |
| Cold start with no feedback                              | Initial picks derive from stated interests; the feedback loop takes over after the first session. |
| Reviewer cannot run the project                          | One-command local setup, sample `.env`, README walkthrough.                                     |

## 11. Future work (Phase 2+)

- Multi-user accounts and authentication.
- Email reminders for saved events.
- Additional cities.
- Mobile / PWA.
- Social features: share saved events, see what friends are attending.
- Smarter ranking signals: weekday vs weekend, price-as-taste, geographic preferences, time-of-day patterns.
- Cloud deployment (course H7; not pursued in MVP since "local only" was a constraint).
